# CoIR (Li et al., 2024 — arXiv 2407.02883)

Skim only: abstract + Section 2-3 methodology via arXiv HTML/PDF. Not a deep read. Stats below are extracted from the arXiv HTML render of Sections 2-3; numeric headline tables not re-verified line-by-line.

## What they measured

Retrieval quality (NDCG-style metrics on a fixed single corpus per task) for 9-10 retrievers across 10 code datasets covering 8 task types and 7 domains. The contribution is breadth of code-retrieval task coverage in a single MTEB/BEIR-compatible package, not any new methodology insight about retrieval-source composition.

Verbatim from the abstract: "COIR comprises ten meticulously curated code datasets, spanning eight distinctive retrieval tasks across seven diverse domains" (https://arxiv.org/abs/2407.02883).

## Did they vary N (concurrent installed retrieval sources/tools)?

**No.** Each query retrieves from a single corpus per task. Authors explicitly note "each query corresponding to exactly one ground-truth corpus" as a stated limitation. There is no N-sweep, no multi-source aggregation, no agent loop. This is a single-retriever-vs-single-corpus benchmark in the BEIR tradition.

## Query construction methodology

Three streams, mixed across the 10 datasets:

1. **Human-authored**: CosQA (20,604 human-annotated web-query/code pairs), StackOverflow QA (user questions + top-voted answers), APPS problem descriptions (from Codewars / AtCoder / Kattis / Codeforces).
2. **LLM-synthesized**: CodeFeedback-ST (single-turn) and CodeFeedback-MT (multi-turn) are "generated by LLMs" per the paper. Synthetic Text2SQL (Gretel) is also LLM-generated (~106k).
3. **Extracted/modified from existing corpora**: CodeSearchNet (GitHub function-docstring pairs, 1M docs across 6 languages), CodeSearchNet-CCR (random function segmentation for code-context retrieval), CodeTransOcean (paired cross-framework / cross-language translations).

**Contamination handling: weak.** The paper does not describe a prospective contamination protocol. Mentions "manual inspection and filtering" but no held-out cutoff date, no post-training-cutoff control, nothing analogous to LiveCodeBench. Several source datasets (CodeSearchNet, APPS, CosQA) predate 2022 and are virtually guaranteed to be in modern model training sets. **This is a real limitation for any reuse in tool-crowding.**

## Retrievers tested

BM25 (sparse baseline); Contriever, E5-Base, BGE-Base, GTE-Base, BGE-M3, E5-Mistral, UniXcoder (dense open-source); OpenAI-Ada-002, Voyage-Code-002 (proprietary). General-purpose embedders dominate; only one code-specific embedder (UniXcoder, Voyage-Code-002).

## What to steal for our code-retrieval query set

| What | How to apply |
|---|---|
| **MTEB/BEIR-compatible schema** | CoIR ships as a pip package with the BEIR data shape. If we adopt the same schema, our queries plug into any existing retriever evaluation pipeline for free. Worth doing. |
| **CosQA (20,604 human-authored web-query/code pairs)** | Closest match to "natural developer query → code answer" of any dataset in CoIR. Candidate query source for our N-sweep if we need single-shot retrieval queries with a known canonical doc. |
| **APPS problem descriptions** | Natural-language problem statements with verifiable code solutions. Adjacent to LiveCodeBench but older and likely contaminated. Use only with post-cutoff filtering. |
| **CodeSearchNet-CCR (code-context retrieval)** | Closest task to what an MCP code-retrieval server would actually be called for: "given this code context, find related code." Worth examining as a query template. |
| **Task taxonomy (text-to-code / code-to-text / code-to-code / hybrid)** | Useful framing for our query set's diversity claim. We should make sure our query set spans at least text-to-code + code-to-code; CoIR's taxonomy is the precedent for that breadth claim. |

## What they didn't measure

- **N as IV.** No sweep over concurrent corpora/tools. Single-corpus per query.
- **Agent in the loop.** No agent decides which retriever to call. Pure retriever-vs-corpus eval.
- **Post-cutoff control.** No LiveCodeBench-style date-filtered subset. Contamination unaddressed.
- **End-task pass@1.** Retrieval quality only, no downstream code-generation correctness. Decoupled from utility.
- **Latency / cost.** Retrievers compared on quality only, not on production cost. Multi-MCP latency invisible.
- **Tool descriptions.** No notion of an agent reading a tool spec and deciding. Same blind spot as CodeRAG-Bench.
- **Cross-corpus aggregation.** Each task's corpus is isolated; no "what if we union 3 corpora and let the retriever decide" variant.

## One open question for tool-crowding

**Is CoIR's CosQA + CodeSearchNet-CCR pair viable as a contamination-acceptable query source for our N-sweep?** CosQA is 2021-era and CodeSearchNet is older still. If we use them, every Claude/GPT/Gemini model in our lineup has likely seen them. The mitigation would be: filter to queries that target post-2024 code, or generate held-out CosQA-style queries against post-cutoff GitHub repos using a CodeSearchNet-style scraper.

Default leaning: (c), because the contamination story is cleaner.

## Compatibility with our gap claim

**Neutral.** CoIR is a retriever-vs-corpus benchmark; it doesn't engage with the agent-decides-which-tool-to-call regime at all. It is neither a precedent we're competing against (different unit of analysis) nor a result that supports our crowding hypothesis (no N-sweep performed). It's a useful source of query-set raw material and a taxonomy reference, nothing more. Citation-worthy in our related-work section as "prior code-retrieval benchmarks fix the corpus and vary the retriever; we fix the task and vary N over installed tools."

4. **Do NOT use CoIR as a methodology precedent for N-sweep claims** — they don't sweep N. Citing them as such would be a misrepresentation.

## Verdict for tool-crowding

**Neutral.** CoIR is the latest in the BEIR/MTEB lineage of single-retriever-vs-single-corpus benchmarks. It does not vary N as an independent variable, does not put an agent in the loop, and does not address post-training-cutoff contamination. It is a useful **source of code-retrieval query templates and a taxonomy reference** but is not a methodology precedent that either supports or contradicts our gap claim. The gap claim ("nobody varies N as an IV") survives this paper intact; CoIR is solidly in the "fixed corpus, varied retriever" camp and so doesn't even attempt the dimension we're measuring.
